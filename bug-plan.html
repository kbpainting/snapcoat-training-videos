#53

Can't add primers to production rates

BugXS

Greg Chung · Elevate Paint Pros · Jul 20

"only products that are labeled 'interior paint' are able to be added. i cannot find a way to access primers or miscellaneous materials without just categorizing them as interior paint"

Root cause — confirmed

client/src/pages/settings/production-rates.tsx:2747-2760. The paint dropdown filters products against the category's estimate type:

if (categoryEstimateType.includes('exterior'))      return productType.includes('exterior');
else if (categoryEstimateType.includes('interior')) return productType.includes('interior');
else if (categoryEstimateType.includes('cabinet'))  return productType.includes('cabinet')
                                                        || productType.includes('interior');

But the product-type dropdown (:3938-3943) offers six types: Interior, Exterior, Primer, Cabinet, Stain, Other. The filter only ever matches interior/exterior/cabinet — so Primer, Stain and Other are invisible in every category. Greg isn't missing a setting; there is no path to them.

Fix

Treat Primer / Stain / Other as universal — they legitimately belong to interior and exterior work. Insert before the category branches:

const UNIVERSAL = ['primer', 'stain', 'other'];
if (UNIVERSAL.some(t => productType.includes(t))) return true;

Single location — I checked for copies of this filter and there is only one. The block at :3337 is help text, not a filter.

Risk

Very low. Purely additive to a dropdown; nothing is removed and no pricing math is touched.

#55

App survives force-quit, phone lags and overheats

BugS–M

Greg Chung · iPhone 12 · Jul 21

"I keep force closing the app but it stays on in the background and causes my phone to lag and overheat. Even when it is force closed"

Root cause — confirmed, and Greg is right

ios/App/App/BackgroundLocationPlugin.swift:120-136. Every available power knob is set to maximum:

desiredAccuracy = kCLLocationAccuracyBest      // full GPS, highest draw
distanceFilter  = 5                            // a callback every 5 metres
allowsBackgroundLocationUpdates = true
pausesLocationUpdatesAutomatically = false     // ← forbids iOS's own battery saver
startUpdatingLocation()
startMonitoringSignificantLocationChanges()    // ← relaunches the app after force-quit

Two of those lines explain the whole report:

  • pausesLocationUpdatesAutomatically = false explicitly disables the built-in optimisation that suspends GPS when the user stops moving. On a parked truck or a job site, the radio keeps running anyway.
  • startMonitoringSignificantLocationChanges() is documented by Apple to relaunch a terminated app. So force-quitting genuinely cannot work.

And it self-perpetuates: locationManagerDidChangeAuthorization (:164-175) restarts tracking whenever authorization is present and isTracking is false. A relaunched process is always fresh, so it immediately re-arms both subscriptions. Kill it, walk 500m, iOS wakes it, and it starts full GPS again — forever.

Running startUpdatingLocation() and significant-change monitoring simultaneously is also redundant; they're two subscriptions doing overlapping jobs.

Fix — staged, safest first
  • 1. Let iOS pause. pausesLocationUpdatesAutomatically = true and set activityType = .automotiveNavigation. Biggest win for the least risk.
  • 2. Loosen precision. kCLLocationAccuracyBest → kCLLocationAccuracyNearestTenMeters, distanceFilter 5 → 25–50m. We report on an interval anyway (reportInterval), so 5m fidelity is thrown away.
  • 3. Only track when there's a reason to. Tracking should be tied to an active shift/appointment, not to app launch. Today authorization alone is enough to start it.
  • 4. Pick one subscription. Continuous updates while on the clock; significant-change only if we truly need post-termination wake — and if we don't, drop it and force-quit starts working like Greg expects.
The advice we gave Greg only half-works. He was told to disable location services. That does stop the drain, but it also silently kills geofencing, on-my-way, and GPS timesheets for him — and per our own notes geofence is already unreliable in the Capacitor app. Fixing the config is the real answer; the workaround quietly disables features he's paying for.
Ships how

Swift change, so it needs a native iOS build — not an OTA web bundle. Worth batching with any other pending native work.

#8

Import production rates from another CRM

FeatureM

Greg Chung · Nov 20 · oldest open item

We did exactly this by hand today. Jason Thomas's 44 rates were built from a DripJobs-format guide and loaded into Paint Pals (co 65). The hard parts are already solved and written down — what's left is turning a script into a screen.
What we learned doing it manually — this is the whole feature
  • DripJobs coat columns are cumulative, ours are per-coat. Their "2 coat" figure is the total speed for a 2-coat job; ours is the speed of the second coat alone. Import naively and every labour line is ~78% high at 2 coats, ~125% at 3.
  • Easy to detect automatically: rates that descend across coats are cumulative; ascending are per-coat.
  • The engine already supports both — [dripjobs:cumulative_coat_rate] in productionRateCalculations.ts:69 — but nothing in the app ever sets it. It's read in 3 files, written in 0. Any customer migrating hits the 78% wall silently.
  • "Each"/price_per_item rows return zero material cost; DripJobs man-hour rows must map to hours_per_item.
  • Linear-foot rates need substrateWidth or material silently under-orders up to 9×.
  • Coverage is clamped to [150,500], which quietly breaks fences (125) and pressure washing (5000).
Build
  • Phase 1 — CSV/paste importer. Upload or paste a rate table → column mapper → preview grid → commit. Auto-detect cumulative vs per-coat and show the call with a sample calculation, letting the user override.
  • Phase 2 — surface the cumulative flag properly. A real "coat rates are cumulative totals" checkbox on the rate editor instead of a magic string in a text field. This is worth doing even if the importer slips.
  • Phase 3 — templates. Ship the PaintPro-style starter set as a one-click "load a starter rate library," which is what most of these customers actually want.
Greg's original ask was a browser script to scrape DripJobs. I'd push back on that — scraping a competitor's UI is fragile and legally murky. Import from an export/CSV instead; same outcome, none of the exposure.

#52

Recurring service / maintenance packages

FeatureL

Greg Chung · Jul 20

"sell maintenance packages for recurring revenue… bill monthly or yearly… have the proposal live in snapcoat, and then have a recurring event on my job schedule every year in the same month for me to assign to a team and generate a work order"

The catch nobody's flagged yet

We have a lot of subscription machinery — company_subscriptions, subscription_plans, subscription_invoices, subscription_events — but all of it bills our customers for SnapCoat, through our Stripe account. Greg is asking to bill his customers on his processor, which is Helcim, not Stripe. None of the existing tables can be reused as-is. This is genuinely net-new.

Decision needed first

Does Helcim support stored-card recurring charges on a per-company basis, and are we willing to hold that liability? Everything else follows from that answer. Three options:

  • A — Full auto-billing. Card on file, we charge on a schedule. Best product, highest risk: card storage, dunning, failed payments, refunds, chargebacks.
  • B — Recurring invoice, manual pay. We generate and send the invoice on schedule; customer pays it like any other. Most of the value, a fraction of the risk — my recommendation for v1.
  • C — Scheduling only. The recurring job lands on the calendar; billing stays manual. Solves the "on my radar" half and is nearly free.
Rough shape (option B)
  • service_plans — company_id, customer_id, source proposal, cadence, price, next_run, status
  • service_plan_occurrences — one row per cycle, linking the generated appointment/work order and invoice; the audit trail and the idempotency guard
  • A scheduled job that materialises the next occurrence, creates the appointment, and raises the invoice. Must be timezone-correct per company and never fire overnight — we've been bitten by both.
  • Proposal flag marking it a recurring plan so the customer-facing document states the cadence and the terms
Biggest hidden cost is the unglamorous part: cancellation, price changes mid-plan, skipped/rescheduled visits, and what happens when a card or invoice fails. Those need answers before the schema is settled, or we'll rebuild it.
